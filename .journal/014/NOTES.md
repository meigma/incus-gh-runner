## 2026-07-20 12:15 — Kickoff
Goal for the session: Review the proposed cross-repository attestation model for proving that builder images originate from authorized GitHub workflows on enrolled physical infrastructure through ephemeral incus-gh-runner VMs, then are admitted by simplestreams-s3.
Current state of the world: incus-gh-runner already owns ephemeral VM lifecycle, exact Incus ownership metadata, hardened host-side operation, and signed release artifacts; simplestreams-s3 already publishes immutable image artifacts and has supply-chain verification building blocks, but the proposed build-session protocol, host attester, admission policy, and verification-summary loop are architectural exploration outside the current security-review slice.
Plan: Check the handoff against the concrete repository architecture and trust boundaries, identify sound parts and missing security/protocol details, and recommend the smallest prototype that can disprove or validate the model before writing a full design.

## 2026-07-20 12:20 — Architecture review
The proposed trust split is directionally correct: the guest may identify candidate bytes, while only the host control plane may assert the source image, VM and host identity, Incus policy state, GitHub assignment, and freshness. The non-circular A-to-B promotion sequence is also sound.

Repository inspection found useful existing seams. incus-gh-runner already resolves and pins the full predecessor fingerprint, hashes profile configuration/devices, materializes that snapshot directly onto each VM, records correlation and source-image metadata, and observes GitHub JobStarted events containing the assigned runner. Its broader read-only CUE validator covers project, network, ACL, profile, storage, listener, and server-policy state. simplestreams-s3 already inspects both split-image components, derives content identities, uploads immutable objects before one conditional index commit, and can therefore place admission immediately before catalog activation.

The handoff needs five corrections before it becomes an implementable security claim:

1. A hot-standby VM exists before a job is assigned, so its session cannot contain GitHub run identity at creation. Create an unbound infrastructure session, then bind it on JobStarted and cross-check the later OIDC token against the scale-set event (repository and immutable IDs, workflow ref/SHA, source SHA/ref, run ID/attempt, event, runner_environment, and exact job identifier). A live spike must determine the precise relationship between the scale-set JobID and OIDC check_run_id.
2. simplestreams-s3 is not an admission authority while the untrusted job retains raw write access to the active S3 prefix. The guest must have only quarantine/staging upload rights; a trusted admission service or worker must hold the sole permission to write immutable active objects and commit streams/v1/index.json.
3. A bearer build-session credential readable by the untrusted job is exfiltratable and does not strongly bind the request to the VM. Keep it in a trusted root-owned guest broker, use an instance-bound channel or host-initiated pull, and persist one-use/idempotent consumption across controller restarts. The signing key must remain host-side.
4. The current profile digest is not the whole Incus security environment. Sign a canonical runtime-manifest digest and the exact CUE policy/baseline digest, rerun the broader validator immediately before signing, and explicitly preserve its assurance boundary: API configuration is not measured runtime enforcement.
5. A root-owned file key proves possession of an enrolled software identity, not a particular physical machine. Treat this as an initial assurance tier; require a non-exportable enrolled TPM key for physical-host identity and separately add quote/PCR evidence if measured boot is claimed.

The provenance should use an in-toto Statement with the current SLSA provenance/v1 predicate: subjects for the metadata archive, QCOW2 disk, and conceptual Incus image fingerprint (SHA-256 over metadata bytes followed by disk bytes); predecessor A in resolvedDependencies; GitHub-controlled inputs in externalParameters; host/controller/runtime facts in trusted platform fields; and a documented custom buildType. A subsequent admission decision should bind the same subjects to an exact policy URI/digest and verifier identity. Do not claim a SLSA level until the complete builder trust base is assessed.

Recommended agile sequence: first capture one real JobStarted event plus OIDC token and prove the correlations; next generate a local host-signed DSSE statement without publishing; then prove quarantine-to-trusted-activation in simplestreams-s3; then issue and consume a signed admission summary by exact fingerprint; only after those seams work add TPM identity, measured boot, SBOM/vulnerability policy, revocation, and fleet enrollment.

## 2026-07-20 12:22 — Subject-binding correction
The handoff's final key rule is too permissive: an untrusted guest that merely reports a digest can obtain provenance for bytes the trusted platform never observed, then supply matching external bytes at publication. The guest may nominate outputs, but a trusted component must independently hash the exact metadata and disk bytes before the host signs, and the admission service must recompute the component digests plus the Incus combined fingerprint before activation. A root-owned helper in trusted predecessor image A, backed by an immutable staging copy or host pull, is sufficient for the first proof; direct trust in job-reported digests is not.

## 2026-07-20 12:32 — Threat-model refinement
The target users are small teams operating a handful of bare-metal Incus hosts, so the enterprise-style trusted artifact-ingestion and byte-observation model is disproportionate. The proposed product boundary is now narrower: incus-gh-runner should attest that one authorized GitHub job ran on an enrolled controller installation, in a particular ephemeral VM, from a pinned predecessor image, under an observed Incus configuration. It should not attest what that job produced or claim that an artifact digest was causally generated inside the VM.

Under this model, an allowlisted GitHub workflow is part of the trusted computing base. The operator owns the risk that the workflow executes hostile pull-request code, mutable actions, compromised dependencies, or malicious build logic. Artifact-to-workflow binding remains the workflow's responsibility through ordinary GitHub artifact attestations or another publication mechanism; the controller emits a separate job-execution receipt. A downstream verifier may join the two on exact run and job identity, but incus-gh-runner does not inspect, hash, stage, publish, or admit artifact bytes.

The remaining controller protocol is intentionally small: create an unbound VM session for hot standby; bind it to the scale-set JobStarted assignment; validate a short-lived GitHub OIDC token against an explicit repository/workflow/ref policy and the assignment; confirm the same active Incus instance UUID, predecessor fingerprint, and configuration digest; then sign an idempotent job-execution receipt with a root-owned host key. TPM identity, measured boot, persistent anti-replay storage, and artifact admission remain optional future assurance tiers rather than prerequisites.

## 2026-07-20 12:40 — TPM support tiers
TPM support should be split into three independently adoptable capabilities rather than treated as one feature.

The recommended first tier is TPM-bound storage for the existing software signing key. Provision the key as a systemd encrypted credential using the host TPM, load it into the controller with `LoadCredentialEncrypted=`, and let the trusted controller sign job-execution receipts normally. This requires no TPM protocol implementation and preserves the current `PrivateDevices=yes` sandbox because the systemd manager performs credential decryption. Initially avoid PCR binding so routine firmware, kernel, and boot changes do not disable the service. This tier prevents copying the encrypted key credential to another machine, but the decrypted key is present in controller memory at runtime and the receipt does not independently prove to a remote verifier that a TPM was used. Describe it as TPM-bound key storage, not TPM-native signing.

The stronger optional tier is a TPM-generated, non-exportable signing key. It requires key provisioning and enrollment, a stable public key identifier, TPM-backed signing in the controller or a small signer helper, rotation and recovery procedures, and access to `/dev/tpmrm0`. The current systemd unit hides physical devices, so direct TPM use would require a deliberate sandbox change or a narrowly scoped signer service over a Unix socket. Public-key enrollment proves possession of the enrolled host key; cryptographically proving that the key originated in genuine TPM hardware additionally requires an attestation-key and endorsement-key enrollment protocol, which is not justified for the first small-team implementation.

Measured boot is a third and substantially larger system: nonce-based TPM quotes, PCR and event-log verification, expected boot-policy management, and an update/recovery process. It should remain out of the initial job-receipt design. If demand appears later, prefer maintainable signed PCR policies over brittle static PCR values.

Agile delivery order: implement and exercise the receipt with a file key; add TPM-bound systemd credential provisioning and prove reboot, cross-host copy failure, rotation, and recovery; add TPM-native signing only if operators require the private key never to leave the TPM; consider measured boot only for a separately stated host-integrity requirement.

## 2026-07-20 13:39 — Reviewed machine-proof design and plan
Created `JOB_MACHINE_PROOF_DESIGN.md` and `JOB_MACHINE_PROOF_IMPLEMENTATION_PLAN.md` as the reviewable architecture and delivery plan. The final scope corrects the earlier OIDC direction: version 1 signs a host-created job/machine receipt from the authenticated `JobStarted` event and the exact JIT runner reference saved before guest delivery. It has no guest request API, OIDC exchange, artifact digest, completion claim, or artifact-admission role.

The direct claim is intentionally narrow: the enrolled controller provisioned one JIT registration into one owned VM, and GitHub later reported a job start for that same registration. Evidence that execution stayed in that VM additionally trusts the approved image/bootstrap not to relay the bearer JIT configuration. A workflow obtains the DSSE envelope through a fixed root-owned guest path and helper, while external consumers use the shipped policy-neutral verifier with an enrolled Ed25519 public key and expected host ID.

Both storage modes feed the same PKCS#8 software key to the controller through a systemd credential. File-backed mode uses `LoadCredential=`; TPM-bound mode uses systemd 250+ `LoadCredentialEncrypted=` with `--with-key=tpm2` and an explicit empty PCR set. The design retains `PrivateDevices=yes`, creates the encrypted credential directory explicitly for older systemd releases, and distinguishes TPM-bound storage from TPM-native signing or measured boot.

Exact dependency pins were rechecked against current module metadata: `go-securesystemslib v0.11.0` for the four named DSSE APIs, existing `actions/scaleset v0.4.0`, existing Incus SDK `v7.2.0`, and Go `1.26.5` standard-library Ed25519/PKCS#8/JSON support. No OIDC or Go TPM library is added.

Two independent review rounds and a final red-team pass corrected the initial drafts. Material fixes include a supported verifier command; a reproducible, non-circular launch digest and existing profile digest; validated JIT response persistence on a stopped VM through conditional Incus `UpdateInstance` plus ETag; restart-safe runner fencing before terminal VM deletion; explicit queue-overflow behavior; immutable committed proof delivery; exact UUID checks; systemd-250-compatible TPM setup and cross-host testing; and a clear host-clock trust statement. The final plan has five gated phases, beginning with local proof format and a thin disposable host-to-VM channel before GitHub and TPM integration.

Validation completed: both JSON examples parse, Markdown fences and whitespace are clean, the three module pins still resolve as latest on 2026-07-20, and the named DSSE APIs were verified in the pinned source. No implementation was started; the plan remains approval-gated.
