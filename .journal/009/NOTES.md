Workflow `29658672280` proved real `systemctl stop`/SIGTERM behavior with an active job: stop completed in 1.61 seconds, the service became cleanly inactive, the job succeeded while the controller was down, and restart replayed completion, deleted the consumed owned runner, retained the replacement, and left an unowned sentinel untouched. A controller-UID-scoped GitHub outage forced the existing broker connection closed; the same process recovered with 1, 2, and 4 second reconnect delays and no systemd restart. Missing image alias, invalid QEMU device, and protected deletion exercises covered create, boot, and delete failures while workflows `29659014770`, `29659260226`, and `29659161928` all succeeded. Hardware revealed that failed create/delete operations retry every reconcile tick, producing a one-second log and API storm. It also showed that an artificially short Incus deadline is unsafe for asynchronous create: the client can time out while Incus continues and leaves a partial instance that contends with later create/delete work. The next bounded phase 6 slice should add operation-failure cooldown/backoff and retain enough operation identity or status to avoid timeout-induced contention.

## 2026-07-18 13:54 — Incus operation backoff merged
